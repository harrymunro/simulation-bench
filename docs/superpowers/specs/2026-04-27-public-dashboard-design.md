# Public Simulation Bench Dashboard — Design

**Status:** Draft for review
**Date:** 2026-04-27
**Owner:** harry
**Related work:** `bd:simulation-bench-tjf`
**Spec target:** v1 of a publicly viewable, read-only dashboard for Simulation Bench, deployed to fly.io.

## 1. Goals & Scope

### Goals

1. Public, read-only window into Simulation Bench: which harness × model combinations score highest on each benchmark, what their approaches look like, and how to reproduce the result.
2. Transparent methodology — the scoring rubric, run protocol, and reviewer forms are reachable in one or two clicks from the leaderboard.
3. Browse each submission's approach (conceptual model, README, source files) inline with syntax highlighting; download bulky logs.
4. Three independent ranking columns on the leaderboard: **Quality / 100**, **Total tokens**, **Wall-clock time**. No combined "meta-score" — operators can rank by any column independently.

### Non-goals (v1)

- Interactive filtering, scatter plots, charts, or dashboards beyond a sortable table.
- User accounts, comments, voting, or any write path.
- Search across submission contents.
- Automatic re-runs of submissions from the dashboard.
- Embedded animations (`animation.gif`/`animation.mp4` are download-only).
- Multi-benchmark navigation logic — the data model supports more benchmarks, but the UI only renders what's actually in `submissions/`.

## 2. Architecture

### Build-time pipeline

```
scores/scores.db ─┐
submissions/      ├─→ harness/build_dashboard.py ─→ dashboard/src/content/  ─→ astro build ─→ dist/  ─→ fly deploy
docs/methodology  ┘     (Python: JSON + per-           (Astro content collections,
                         submission MD stubs)            pages, layout)
```

Python owns the data layer (where `scores.db` and submission folders already live). Astro owns presentation. The handoff between them is a flat pile of JSON + Markdown — no Python ↔ Node coupling beyond filesystem reads.

### Repository additions

```
dashboard/                          # Astro project (new)
  astro.config.mjs
  package.json
  src/
    content/
      submissions/                  # one .md per submission, generated
      methodology/                  # generated copies of root SCORING_GUIDE.md, RUN_PROTOCOL.md
    data/
      leaderboard.json              # generated by build_dashboard.py
    pages/
      index.astro                   # leaderboard
      methodology/index.astro
      methodology/scoring.astro
      methodology/protocol.astro
      methodology/dashboard.astro
      submissions/[id].astro        # submission overview
      submissions/[id]/[...file].astro  # syntax-highlighted file viewer
    layouts/
    components/
  public/
    submissions/<id>/event_log.csv  # large download-only files, copied at build time

fly.toml                            # serves dist/ via caddy
Dockerfile                          # caddy:2-alpine + COPY dashboard/dist /srv
Caddyfile

harness/
  build_dashboard.py                # new: scores.db + submissions/ → Astro inputs
  normalize_tokens.py               # new: backfills token_usage.json + run_metrics.json

docs/
  superpowers/specs/2026-04-27-public-dashboard-design.md   # this document
```

### Build-time data flow

1. `harness/record_score.py --from-json scores/seed_scores.json` refreshes `scores.db` from the JSON source of truth.
2. `harness/build_dashboard.py` queries `scores.db` (the `leaderboard` view plus per-category columns) → writes `dashboard/src/data/leaderboard.json`.
3. For each submission folder, `build_dashboard.py` walks the file tree, classifies entries against an allowlist + size threshold, and emits:
   - `dashboard/src/content/submissions/<id>.md` with frontmatter (decoded folder, scores, file manifest).
   - Copies of large files to `dashboard/public/submissions/<id>/`.
4. `build_dashboard.py` also copies `SCORING_GUIDE.md` and `RUN_PROTOCOL.md` into `dashboard/src/content/methodology/`.
5. `astro build` consumes those inputs → produces `dashboard/dist/`.
6. `fly deploy` ships a Caddy container with `dashboard/dist/` baked into the image.

## 3. Data Model & Token/Time Normalization

### Protocol changes

`RUN_PROTOCOL.md` §4 (`run_metrics.json`) and §5 (`token_usage.json`) change from **"if available"** to **required**. Both files must exist for every new submission. Both still permit `null` values when the platform doesn't expose the data, but the file must exist with `token_count_method: "exact" | "reported" | "estimated" | "unknown"` set explicitly.

### Canonical schemas (already in protocol; now enforced)

```json
// token_usage.json
{
  "input_tokens": 78000,
  "output_tokens": 21000,
  "total_tokens": 99000,
  "token_count_method": "reported",
  "estimated_cost_usd": null
}
```

```json
// run_metrics.json (subset relevant to the dashboard)
{
  "command": "python sim.py",
  "runtime_seconds": 297.0,
  "return_code": 0,
  "timed_out": false,
  "python_loc": {...},
  "file_counts": {...}
}
```

### Backfill

One-time script `harness/normalize_tokens.py` writes `token_usage.json` and `run_metrics.json` into the four existing submissions:

| Submission | `token_usage.json` | `run_metrics.json` |
| --- | --- | --- |
| claude-code/opus-4-7 | `total_tokens: 116900`, split null, `method: "reported"` | `runtime_seconds: 699` |
| codex-cli/gpt-5-5 | `total_tokens: 503000`, split null, `method: "reported"` | `runtime_seconds: 400` |
| pi-agent/gemini | `input_tokens: 78000`, `output_tokens: 21000`, `total_tokens: 99000`, `method: "reported"` | `runtime_seconds: 297` |
| gsd2/gemini | All null, `method: "unknown"` | `runtime_seconds: null` |

Source for the backfill: the informal `time_s`, `tokens`, `tokens_in`/`tokens_out` fields in each submission's `submission.yaml`. Those YAML fields stay in place for now (no destructive edit) — they are simply superseded by the canonical files going forward.

### Scoreboard DB extensions

Additive `ALTER TABLE` statements on `scores.db.submissions`:

```sql
ALTER TABLE submissions ADD COLUMN total_tokens          INTEGER;
ALTER TABLE submissions ADD COLUMN input_tokens          INTEGER;
ALTER TABLE submissions ADD COLUMN output_tokens         INTEGER;
ALTER TABLE submissions ADD COLUMN token_count_method    TEXT;
ALTER TABLE submissions ADD COLUMN runtime_seconds       REAL;
ALTER TABLE submissions ADD COLUMN intervention_category TEXT;
```

`harness/record_score.py --from-json` is updated to read each submission's `token_usage.json` + `run_metrics.json` at ingest time and populate these columns. The DB stays the single query surface for the dashboard build.

`harness/scores_db.py` schema constant gets the new columns; the `leaderboard` view is updated to include them.

### Intervention category — structured field

Protocol §8 currently asks operators to describe human interventions narratively in `README.md` or a free-form `interventions.md`. To support a leaderboard badge, this design adds a structured field to `submission.yaml`:

```yaml
intervention:
  category: autonomous | hints | manual_repair | failed | unrecorded
  notes: "free text; references to interventions.md welcomed"
```

`record_score.py` reads `submission.yaml.intervention.category` into the new DB column; missing or unparseable → `unrecorded`. The narrative `interventions.md` (or the README section) remains the authoritative description; the structured field exists only to drive the badge.

### Skill updates

- `create-submission`: scaffold empty stub `token_usage.json` and `run_metrics.json` files alongside the standard layout.
- `evaluate-submission`: warn loudly (not fatal) when either file is missing or `token_count_method` is `"unknown"`.

## 4. Pages & UX

### Site map

```
/                                   Leaderboard (one row per scored submission)
/methodology/                       Index of methodology docs
/methodology/scoring/               Renders SCORING_GUIDE.md
/methodology/protocol/              Renders RUN_PROTOCOL.md
/methodology/dashboard/             "How this dashboard is built" (data sources, build, deploy)
/submissions/<id>/                  Submission overview (decoded folder, scores, file index)
/submissions/<id>/<file-path>       Single file viewer (syntax-highlighted)
```

### Leaderboard (`/`)

Single sortable HTML table. Columns:

| # | Date | Benchmark | Harness | Model | Tag | Quality | Tokens | Time | Intervention |

- **Quality** — `total_score / 100`. On hover or row-expand: the seven category sub-scores.
- **Tokens** — `total_tokens`, formatted (`503k`); shown as `—` if null. Hover tooltip shows input/output split + a small badge for `token_count_method`.
- **Time** — `runtime_seconds`, formatted (`4m 57s`); `—` if null.
- **Intervention** — badge derived from protocol §8: ✅ autonomous · ⚠️ hints · 🛠 manual repair · ❌ failed · `?` unrecorded.
- Each row links to `/submissions/<id>/`.
- Sorting is client-side (a small Astro island). Default sort: Quality desc.

### Submission overview (`/submissions/<id>/`)

- Header: decoded folder name + scores card (seven categories + total).
- Sections:
  - Conceptual model (rendered)
  - README (rendered)
  - Reviewer form (rendered)
  - Evaluation report (selected fields rendered, full JSON linked)
  - Source code: file tree linking each text file to the file viewer
  - Downloads: large files (event log, animations) with size labels

### File viewer (`/submissions/<id>/<file-path>`)

- Shiki syntax highlighting; language inferred from extension.
- Breadcrumb back to the submission overview.
- "View raw on GitHub" fallback link for users who want byte-for-byte verification.

### Methodology pages

Straight Markdown rendering of `SCORING_GUIDE.md` and `RUN_PROTOCOL.md`, plus a hand-written "Dashboard data sources" page that explains the build pipeline at a high level: scores from `scores.db` (sourced from `seed_scores.json`), tokens from `token_usage.json`, time from `run_metrics.json`. Every column header on the leaderboard links to the methodology section that defines it.

### Client-side state

Sort only. No filtering UI, no search bar, no charts. Add later if there is demand.

## 5. Build, Deploy & Security

### Local build chain

```bash
# 1. Refresh DB from JSON seed (and any new token_usage.json / run_metrics.json)
python harness/record_score.py --from-json scores/seed_scores.json

# 2. Generate Astro inputs from DB + submissions/
python harness/build_dashboard.py

# 3. Build static site
cd dashboard && npm run build      # → dashboard/dist/

# 4. Deploy
fly deploy
```

A top-level `Makefile` wraps these into:

```makefile
make dashboard   # steps 1–3
make deploy      # steps 1–4
```

### Container & Fly configuration

- `Dockerfile`: `caddy:2-alpine` + `COPY dashboard/dist /srv` + `COPY Caddyfile /etc/caddy/Caddyfile`.
- `Caddyfile`: `file_server`, gzip, sane cache headers, basic security headers.
- `fly.toml`: single `shared-cpu-1x` machine, one region, auto-stop machines (free-tier friendly), HTTPS handler. No volumes — site is fully baked into the image.
- App name: `simulation-bench` (or whatever is available; default `<app>.fly.dev` URL is fine for v1).

### Security posture

The repository is public, so the dashboard inherits the repo's threat model rather than introducing a new one. Key properties:

1. **Read-only.** No backend, no DB at runtime, no auth surface.
2. **Public-by-design data.** The site only ships content already in the public repo: `scores/seed_scores.json`, `submissions/<folder>/` artefacts, methodology docs.
3. **No secrets in the deploy.** No env vars, no API keys baked into the image. The Dockerfile is small enough to audit by inspection.
4. **Caddy hardening defaults**: HSTS, no directory listing, drop server header, basic CSP (`default-src 'self'`; allow inline styles for Shiki highlighting).
5. **Explicit allowlist for shipped artefacts.** `build_dashboard.py` only copies files matching the protocol's required + optional deliverables (plus `results/`). Anything else in a submission folder — stray `.env`, scratch notes, IDE configs — is ignored. This protects against an operator accidentally checking secrets into a submission folder.

### What is NOT in the deploy

- `.beads-credential-key`
- `.dolt/`
- Any file outside `dashboard/dist/`

Verified by inspecting the Dockerfile — it only `COPY`s `dashboard/dist` and `Caddyfile`.

## 6. Rollout Plan

Five phases. Each is independently shippable; we can stop or pause between any two without leaving the project broken.

### Phase 1 — Protocol & data normalization (no UI yet)

- Update `RUN_PROTOCOL.md`: `token_usage.json` and `run_metrics.json` required.
- Write `harness/normalize_tokens.py`; backfill the four existing submissions.
- Backfill `run_metrics.json` from `submission.yaml.time_s` for the three submissions that have it.
- Extend `scores.db` schema (new columns).
- Update `harness/record_score.py` to read token/time at ingest and write them into the DB.
- Update `create-submission` and `evaluate-submission` skills.
- **Exit criterion:** `python harness/scoreboard.py --detailed` shows tokens + time columns for all four existing submissions; `gsd2` shows `—` for both.

### Phase 2 — Dashboard scaffold (local only)

- `npm create astro@latest dashboard` (minimal template).
- `harness/build_dashboard.py` v0: writes `leaderboard.json` + per-submission stub `.md` files.
- Astro: leaderboard page renders the table; methodology pages render protocol + scoring docs.
- **Exit criterion:** `npm run dev` shows leaderboard + methodology locally with all four submissions; sorting works.

### Phase 3 — Submission browser

- File-walker in `build_dashboard.py` (allowlist + size threshold).
- Astro dynamic routes `[id].astro` and `[id]/[...file].astro` with Shiki highlighting.
- Copy large files to `dashboard/public/submissions/<id>/` for download.
- **Exit criterion:** every submission page lists its files; clicking a `.py` shows highlighted code; clicking `event_log.csv` downloads it.

### Phase 4 — Deploy to fly.io

- `Dockerfile`, `Caddyfile`, `fly.toml`.
- `fly launch` (one-time), then `fly deploy`.
- DNS / custom domain optional (default `<app>.fly.dev` is fine to start).
- **Exit criterion:** public URL serves the dashboard; methodology page links resolve; downloads work.

### Phase 5 — Polish & documentation

- Top-level `Makefile` wrapping build + deploy.
- README section: "How to update the dashboard" (3 commands).
- Methodology page: "Dashboard data sources" explaining DB → JSON → site flow.
- **Exit criterion:** a new operator can add a submission + score and ship a refreshed dashboard in <10 minutes following the README.

### Sequencing rationale

Phase 1 is independent of the dashboard — it improves the data layer regardless of UI fate, and it pays off the first technical-debt item (token capture inconsistency) immediately. Phases 2 → 3 → 4 are linear: no UI without the scaffold, no deploy of a half-done UI. Phase 5 is a cleanup pass once the site is live.

## 7. Open questions / future work

These are explicitly out of v1. Logged here so they don't get lost.

- **Multi-benchmark UX.** When benchmark 002 lands, the leaderboard needs a benchmark filter or one tab per benchmark. The data model already supports this; only the UI needs work.
- **Cost overlay.** `token_usage.json` has an optional `estimated_cost_usd` field. Once we have prices for each model × token-type, render a "$" column.
- **Score history.** `scores.scores` allows multiple reviews per submission via `(submission_id, reviewer, review_date)`. The dashboard currently shows "latest score per submission"; eventually it could show review history per submission.
- **Pareto frontier visualisation.** A scatter plot of tokens vs. time, with quality as colour/size, would be a useful secondary view on the methodology page. Astro island, low priority.
- **Search.** Lunr/Pagefind across submission READMEs and conceptual models. Defer until there are >10 submissions.
